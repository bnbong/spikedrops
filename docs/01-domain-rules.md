# SpikeDrops Pay Domain Rules

## 1. 판매 / 드롭 규칙

1. 판매 시작 전에는 주문할 수 없다.
2. 판매 종료 후에는 주문할 수 없다.
3. `CLOSED` 또는 `CANCELED` 상태의 `DropEvent`에서는 주문할 수 없다.
4. `ON_SALE` 상태가 아닌 `DropItem`은 주문할 수 없다.
5. 사용자의 주문 수량은 `DropItem.purchaseLimitPerUser`를 초과할 수 없다.
6. 동일 상품이라도 `DropEvent`가 다르면 판매 가격, 판매 기간, 구매 제한, 재고가 다를 수 있다.

## 2. 장바구니 규칙 - v1

1. 사용자는 하나의 `ACTIVE` 장바구니를 가질 수 있다.
2. 장바구니는 `CartItem`을 소유한다.
3. 장바구니에 상품을 담는 행위는 재고를 예약하지 않는다.
4. 같은 `DropItem`을 장바구니에 다시 담으면 기존 `CartItem`의 수량을 증가시킨다.
5. `CartItem.quantity`는 1 이상이어야 한다.
6. `CartItem.quantity`는 `DropItem.purchaseLimitPerUser`를 초과할 수 없다.
7. 장바구니 상품은 checkout 시점에 판매 가능 여부, 가격, 재고를 다시 검증한다.
8. `CHECKED_OUT`, `EXPIRED`, `ABANDONED` 상태의 장바구니는 수정할 수 없다.
9. checkout이 성공하면 장바구니는 `CHECKED_OUT` 상태가 된다.
10. checkout이 실패하면 장바구니는 주문으로 변환되지 않는다.

## 3. 주문 / 재고 규칙

1. 재고가 부족하면 주문할 수 없다.
2. 주문 생성 시 재고는 판매 확정이 아니라 예약 상태가 된다.
3. 주문 생성 시 `Inventory.availableQuantity`는 감소하고 `reservedQuantity`는 증가한다.
4. 결제 성공 시 예약 재고는 판매 확정 상태가 된다.
5. 결제 성공 시 `Inventory.reservedQuantity`는 감소하고 `soldQuantity`는 증가한다.
6. 결제 실패, 주문 취소, 주문 만료 시 예약 재고는 복구된다.
7. 예약 재고 복구 시 `Inventory.reservedQuantity`는 감소하고 `availableQuantity`는 증가한다.
8. 재고 수량은 음수가 될 수 없다.
9. `totalQuantity = availableQuantity + reservedQuantity + soldQuantity`는 항상 성립해야 한다.
10. 결제 완료된 주문은 만료 처리될 수 없다.

## 4. 주문 규칙

1. 주문은 하나 이상의 `OrderLine`을 가진다.
2. `OrderLine`은 주문 당시의 상품명, 판매가, 수량을 snapshot으로 보관한다.
3. 상품명이 변경되어도 이미 생성된 주문의 `OrderLine` snapshot은 변경되지 않는다.
4. 주문 만료 시간은 주문 생성 시점에 결정된다.
5. `PAID`, `EXPIRED`, `CANCELED`, `REFUNDED` 상태는 종료 상태로 간주한다.

## 5. 결제 규칙

1. 하나의 주문에는 하나의 결제만 연결된다.
2. 결제 승인 요청은 멱등하게 처리되어야 한다.
3. 같은 결제 승인 요청이 여러 번 들어와도 결제 결과는 한 번만 반영된다.
4. 승인 완료된 결제는 실패 상태로 되돌릴 수 없다.
5. 실패로 확정된 결제는 승인 완료 상태로 변경할 수 없다.
6. 결제 처리 중인 주문은 기본 만료 처리 대상에서 제외한다.

## 6. 환불 규칙 - v1 이후

1. 환불은 결제 완료된 주문에 대해서만 요청할 수 있다.
2. 환불 요청은 멱등하게 처리되어야 한다.
3. 같은 환불 요청이 여러 번 들어와도 환불 결과는 한 번만 반영된다.
4. 환불 완료된 주문은 다시 결제 완료 상태로 되돌릴 수 없다.
5. 환불 완료 시 금전 기록이 남아야 한다.

## 7. 정산 / 원장 규칙 - v1 이후

1. 결제 승인과 환불 완료는 금전 기록으로 남긴다.
2. 확정된 금전 기록은 직접 수정하지 않는다.
3. 취소나 보정이 필요하면 별도의 보정 기록을 추가한다.

## 8. 일관성 지원 규칙 - v1 이후

1. 결제 승인, 환불 완료, 정산 생성 이후 후속 작업은 이벤트로 분리할 수 있다.
2. 후속 작업 실패는 재시도 가능해야 한다.
3. 후속 이벤트는 중복 발행될 수 있으므로 소비자는 멱등하게 처리해야 한다.
